> 角色：高级交易风险与执行专家 | 输出：结构化执行方案设计与风险矩阵

## 1. 回测 -> 实盘差距清单与缓解方案

### 1.1 核心差距概览

回测与实盘之间的差距是量化交易中最持久且最昂贵的挑战。研究显示，**绝大多数在回测中表现良好的策略在实盘部署后会失败**。以下是针对多周期滤波策略的系统性差距分析。

### 1.2 差距维度

| # | 差距类别 | 策略具体风险 | 严重程度 | 缓解方案 |
|---|---|---|---|---|
| **G1** | **过拟合** | 跨4周期参数联合优化，自由度极高，容易记忆噪声而非模式 | 致命 | 引入滚动交叉验证(WFA)、Deflated Sharpe Ratio、PBO指标；寻找参数**平台区**而非尖峰 |
| **G2** | **信号重绘(Repainting)** | `np.gradient`计算依赖`close`和`high`——若在bar未关闭时计算，会引入未来数据 | 致命 | **关键检查**：确认`np.gradient`仅使用已关闭bar的数据；实盘中只能在bar关闭后触发信号 |
| **G3** | **前向预测不可靠** | 8bar前向预测在实盘中会随新数据不断漂移，历史预测点会被修正 | 高 | 前向预测仅作为**参考止损轨迹**而非信号依据；每bar结束时固定该bar的预测值(不追溯修改) |
| **G4** | **执行假设不现实** | 回测假设中间价成交、零滑点、瞬时成交 | 高 | 买入按卖一价+1tick、卖出按买一价-1tick建模；滑点预估1.5-2倍典型值 |
| **G5** | **换仓成本低估** | 港股T+0可能日内多次进出，累积佣金+印花税+滑点 | 中-高 | 计入港交所印花税(0.13%)、证监会征费(0.0027%)、交易征费(0.00565%)、经纪佣金 |
| **G6** | **数据偏差** | 历史数据不含幸存者偏差；使用的前复权价格与实盘不一致 | 中 | 使用时点成分股列表(PiT)；复权方式一致性检查 |
| **G7** | **市场体制变化** | 策略只在特定市场体制(Beta/Vol regime)中有效 | 中 | 分年度测试；在不同波动率/趋势体制下分别测；设置体制过滤器 |
| **G8** | **样本量不足** | 4周期滤波信号稀疏，可能总交易次数<100 | 中 | 要求最低30笔，优选100+笔交易才能确认显著性 |

### 1.3 关于 np.gradient 和信号重绘的专项分析

**核心问题**：`np.gradient` 是否含未来函数？

`np.gradient` 是中心差分计算。对于序列 `[x0, x1, x2, ..., xn]`：

- `gradient[0] = x1 - x0`（前向差分，用到未来1个点的数据）
- `gradient[i] = (x_{i+1} - x_{i-1}) / 2`（中心差分，**用到前后各1个点的数据**）
- `gradient[n] = xn - x_{n-1}`（后向差分）

**如果在bar还未close时对`close`数组执行gradient，则当新数据到来时，历史gradient值会变化。** 这就是信号重绘。

**缓解方案**：
1. 只在bar关闭后(即该bar的最高价/最低价/收盘价确定后)计算gradient
2. 使用 `close_shifted = close.shift(1)` 后再计算gradient，确保只使用已确定数据
3. 实盘中维护延迟一个bar的数据缓冲区，只对缓冲区做计算
4. 验证方法：bar replay模式下逐步执行，观查已关闭bar的gradient值是否变化

### 1.4 前向预测在实盘中的不可靠性

策略要求前向预测8bar作为止损参考，这需要更严格的管理：

- **预测漂移**：每个新bar到来时，前8bar的预测值会被修正，导致止损参考线不断变化
- **解决方案**：每bar结束时"冻结"该bar上的预测值；后续该bar的预测值不再修改
- **实盘行为**：止损参考是一个逐步冻结的钟形曲线，而非回测中看到的"完美"连续曲线

## 2. 实盘执行架构建议

### 2.1 整体架构：混合事件驱动 + 定时任务

```
+------------------------------------------+
|              监控 & 告警层                  |
|  (Telegram/Slack/邮件 + Grafana面板)       |
+------------------------------------------+
           ↑ 指标推送 ↑
+------------------------------------------+
|           风险管理中间件                    |
|  (熔断器 + 仓位校验 + 滑点缓冲区)           |
+------------------------------------------+
           ↑ 过滤后信号 ↑
+------------------------------------------+
|          策略引擎(事件驱动)                 |
|  - Schmitt触发逻辑                        |
|  - 4周期对齐                              |
|  - 前向预测计算(bar结束时触发)             |
|  - 分段混合规则(保护期 -> 跟踪期)           |
+------------------------------------------+
      ↑ OHLCV ↑        ↑ 信号 ↓
+------------------------------------------+
|          数据层 (WebSocket + 定时器)         |
|  15min | 60min | 日线 | 周线               |
|  WebSocket为主，每5min定时校验              |
+------------------------------------------+
      ↑ 原始数据  ↑
+------------------------------------------+
|          数据源(港股实时行情API)            |
+------------------------------------------+
      ↓ 执行 ↓
+------------------------------------------+
|          交易执行层                        |
|  (券商API / OMS订单管理)                  |
+------------------------------------------+
```

### 2.2 关键设计决策

| 决策项 | 推荐方案 | 理由 |
|---|---|---|
| **数据获取方式** | **WebSocket + 定时轮询双通道** | WebSocket提供低延迟推送；定时轮询(每5min)作为保底，防止断连漏数据 |
| **跨周期对齐策略** | 最小公倍数触发 + 状态缓存 | 15min=900s；60min=3600s；日线=86400s；周线=432000s；在较短周期bar关闭时检查较长周期是否有新bar关闭 |
| **信号触发时机** | **仅在bar关闭时触发** | 避免信号重绘；所有4个周期都需要等待各自的bar关闭 |
| **Bar状态管理** | 每个周期维护 `is_closed` 标志 | 定时器校验；WebSocket推送确认；DFS(数据新鲜度评分) |
| **保存中间状态** | 每min持久化策略状态到本地DB | 异常重启后可恢复，避免状态丢失 |
| **前向预测冻结** | 每个bar结束时冻结该bar上的预测值 | 预测值作为参考止损轨迹，不回溯修改 |

### 2.3 延迟预算

```
数据接收 (WebSocket/API)          ~50-200ms
数据清洗 & 对齐                    ~5-20ms
策略计算 (4周期 + Schmitt + 梯度)  ~30-100ms
前向预测计算                       ~20-50ms
风险管理校验                       ~5-10ms
订单提交 (API)                    ~50-200ms
-----------------------------------------
总延迟预算                        ~160-580ms (可接受范围)
```

### 2.4 事件驱动 vs 定时轮询

| 对比维度 | 事件驱动 | 定时轮询 | 推荐方案 |
|---|---|---|---|
| 延迟 | 毫秒级推送 | 取决于轮询间隔 | 两者结合 |
| 实现复杂度 | 高(状态机/重连/排序) | 低(简单循环) | 先轮询上线，后升级 |
| 可靠性 | 依赖连接稳定性 | 简单可控 | 互备 |
| 资源消耗 | 低(空闲时静止) | 固定CPU | - |
| 调试难度 | 高 | 低 | 适合分期 |

**推荐路径**：  
- **阶段1(最小可行产品MVP)**：基于定时轮询(10秒间隔) + 简单队列
- **阶段2(稳定期)**：引入WebSocket事件驱动作为主要通道，轮询降为保底
- **阶段3(生产级)**：完整的异步事件总线架构

## 3. 风险控制矩阵（量化指标）

### 3.1 层级风险控制

```
层级1: 策略级风控 ─── 每笔交易级别
  ├── 单笔止损(固额比例或前向预测止损轨迹)
  ├── 单笔止盈(保护期 + 跟踪期)
  └── 单笔最大仓位(% of capital)

层级2: 日内风控 ─── 每个交易时段
  ├── 日内最大亏损限额
  ├── 日内最大交易次数
  ├── 日内最大换手率
  └── 日内最大敞口

层级3: 组合风控 ─── 滚动窗口
  ├── 最大回撤限度(硬性熔断)
  ├── 连续亏损次数熔断
  ├── 波动率自适应仓位调整
  └── 跨周期信号冲突解决

层级4: 系统风控 ─── 全局
  ├── 数据断连检测与暂停
  ├── 异常订单率熔断
  └── 手动紧急停止(Kill Switch)
```

### 3.2 量化指标矩阵

| 风险指标 | 计算公式 | 阈值(建议) | 触发动作 | 备注 |
|---|---|---|---|---|
| **单笔止损** | 入场价 x (1 - 止损%) 或 预测止损轨迹 | 账户的0.5-1.5% | 市价止损单 | 前向预测止损轨迹动态计算 |
| **保护期止损** | 入场后N分钟内固定止损 | N=策略保护周期长度 | 过渡到跟踪期 | 保护期内禁用跟踪止损 |
| **日内最大亏损** | 当日累计PnL(含未实现) ≤ 限额 | 账户的3-5% | 暂停当日交易 | 港交所T+2结算，需考虑未实现 |
| **日内最大交易次数** | 累计已成交订单数 | 视策略频率定 | 暂停入场 | 防止系统故障导致过度交易 |
| **最大回撤(硬熔断)** | 峰值到谷值跌幅 | 8-12% | 平仓并暂停策略 | 需要人工复核后重启 |
| **连续亏损熔断** | 连续N笔亏损交易 | 5-8笔 | 暂停策略并报警 | 可能涉及市场体制变化 |
| **波动率自适应** | 当前ATR / N日ATR中位数 | >2.0倍 | 仓位减半 | 保护期不适用 |
| **信号冲突率** | 4周期信号不一致比例 | >60% | 暂停入场 | 多周期信号分歧加大 |
| **滑点预估** | 成交价 vs 触发价 | 超过N个tick | 报警并调整 | 港股市况不同差异大 |
| **数据断连** | 超过N秒无新数据 | 60秒 | 暂停交易并报警 | WebSocket+轮询互备 |
| **异常订单率** | 撤销/修改/拒绝占订单数比例 | >20% | 暂停并检查系统 | 券商API问题信号 |

### 3.3 熔断器状态机

```
        +----------+
        |  正常运作  | <────────────── 人工恢复
        +----+-----+                    ↑
             | 触发风控条件                |
             v                            |
        +----------+                 +----+------+
        |  触发熔断  | ──冷却期(N分钟)─> | 冷却阶段  |
        +----------+                 +-----------+
             |                              |
             | 紧急情况                       | 人工审核
             v                              v
        +----------+                 +----------+
        |  紧急平仓  |                 | 人工恢复  |
        +----------+                 +----------+
```

**关键规则**：
- 熔断器独立于策略信号逻辑运行，**不能被策略绕过**
- 所有触发事件记录详细信息用于事后分析
- 熔断后**不自动恢复**，需人工审核后手动重启
- 冷却期不允许开新仓，但允许关闭现有头寸

### 4.1 实时监控面板

| 类别 | 指标 | 刷新频率 | 告警阈值 | 告警方式 |
|---|---|---|---|---|
| **性能监控** | 当前PnL(已实现+未实现) | 每tick | - | 数值显示 |
| | 日内PnL | 每分钟 | < -3% | 黄色告警 |
| | 当前持仓 | 每tick更新 | - | 数值显示 |

| | Schmitt触发状态(上次触发时间) | 持续监控 | 长时间无触发 | 黄色告警 |

| | 交易频率 vs 回测期 | 每日 | 差距 > 30% | 黄色告警 |
| | 持仓时长分布 vs 回测期 | 每周 | 显著差异 | 黄色告警 |

### 4.2 策略漂移检测

策略漂移(Strategy Drift)是实盘中最容易忽略的问题。建议实施以下检测机制：

1. **滚动窗口绩效对比**：每30笔交易计算一次滚动Sharpe、胜率、平均盈亏比，与回测期对比
2. **KS检验**：对实盘和回测的每日PnL分布做Kolmogorov-Smirnov检验，p<0.05时告警
3. **PSI(Population Stability Index)**：监控信号值分布的变化，PSI>0.1时提示警告，>0.25时告警
4. **体制变化检测**：监控基准指数波动率、相关性结构的变化
5. **Alpha衰减曲线**：跟踪因子有效性随时间的变化趋势

## 5. 港股特殊考虑

### 5.1 交易规则

| 规则项 | 说明 | 对策略的影响 |
|---|---|---|
| **T+0** | 当日可多次买卖 | 策略可日内反复进出，但需控制交易成本 |
| **T+2结算** | 交易后第2个工作日结算 | 当日现金流不受限，但需预留结算资金 |
| **无涨跌停** | 个股无涨跌幅限制 | **无熔断保护**，极端行情下单日波动可达50%+ |
| **开收市竞价** | 开市前时段(09:00-09:20)和收市竞价(16:00-16:10) | 信号触发需避开竞价时段，否则成交价不可控 |
| **最小变动价位** | 不同价位股票不同(0.001/0.005/0.01/0.02/0.05/0.10/0.20/0.50/1.00) | 止损/止盈设置需基于对应tick size |
| **印花税** | 成交金额的0.13%(买卖双向) | 高额交易成本，对短周期策略影响显著 |
| **交易征费+证监会征费** | 0.00565% + 0.0027% | 额外摩擦成本 |

### 5.2 港股数据源注意事项

| 数据源类型 | 延迟 | 可靠性 | 成本 | 推荐程度 |
|---|---|---|---|---|
| 免费Level 1(15分钟延迟) | 15-18min | 低 | 免费 | 不适用实时交易 |
| 付费Level 1(实时) | ~200-500ms | 中 | ~200HKD/月 | 入门级 |
| 付费Level 2(深度行情) | <50ms | 高 | ~500-1000HKD/月 | 推荐 |
| 券商API实时行情 | ~100-500ms | 中-高(取决于券商) | 免费(开户后) | 推荐用于MVP |
| 第三方行情数据商(如Wind/同花顺) | ~50-200ms | 高 | 高 | 机构级 |

**注意**：港股实时数据需购买订阅，免费API通常提供15分钟延迟数据，**不适合实时交易策略**。

### 5.3 港股执行建议

1. **优先使用券商API**的直接行情源，避免第三方中间层增加延迟
2. **避开竞价时段**执行策略的主要信号逻辑（开市前09:00-09:20、收市竞价16:00-16:10）
3. **假突破防范**：港股市值集中度高，流动性不均，中小盘股易出现假突破触发Schmitt信号
4. **股息调整**：港股除息日价格跳空会触发策略的止损/止盈，需提前标记除净日
5. **台风/黑雨**：港股有恶劣天气停市机制，需加入日历事件处理逻辑
6. **北向/南向通**：若涉及沪深港通，注意交易日差异和额度限制

## 6. 分阶段上线推荐路径

### 阶段1：验证与修复（2-4周）

```
目标：确认策略逻辑在实盘环境下有效，没有重绘/未来函数问题

[  ] 1. 审计所有信号计算逻辑，确保不使用未来数据
     - 检查 np.gradient 实现
     - 检查 bar 关闭前是否触发了计算
     - 检查前向预测的"冻结"机制
[  ] 2. 回测框架增加滑点模型（买按卖一+1tick，卖按买一-1tick）
[  ] 3. 回测框架增加交易成本模型（含印花税 + 征费 + 佣金）
[  ] 4. 修复后重新回测，确认策略仍有效
[  ] 5. 启动Bar Replay模式验证信号一致性
[  ] 6. 部署Paper Trading(模拟交易)环境
```

### 阶段2：模拟交易（4-8周）

```
目标：在实盘环境中验证执行能力和策略适应性

[  ] 1. 接入券商模拟账号/Paper Trading API
[  ] 2. 部署数据层（WebSocket + 定时校验，4周期对齐）
[  ] 3. 部署策略引擎（定时轮询模式，bar关闭时触发）
[  ] 4. 部署基础执行模块（限价单为主）
[  ] 5. 实现基础风控（日内最大亏损 + 单笔止损）
[  ] 6. 搭建监控面板（Grafana基础面板）
[  ] 7. 运行4周以上的模拟交易
[  ] 8. 收集实盘与回测差距数据，调整滑点/成本参数
```

### 阶段3：小资金实盘（4-8周）


### 阶段4：正常规模实盘（长期）

```
目标：逐步扩大资金规模，优化执行

[  ] 1. 逐步增加资金（每次25%，观察1-2周再继续）
[  ] 2. 升级到事件驱动架构（WebSocket为主）
[  ] 3. 引入高级风控（波动率自适应 + 体制检测）

[  ] 5. 引入策略漂移检测
[  ] 6. 建立定期的策略再评估机制（月/季度review）
[  ] 7. 建立回测-实盘差距的持续记录和改进闭环
```
