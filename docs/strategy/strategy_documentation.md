## 1. 系统概述

本系统是一个**配置驱动的多时间框架量化过滤策略研究平台**，基于 Streamlit 构建。核心能力：

- **4 视图 2×2 布局**：同时对比 4 个独立配置的过滤策略
- **8 时间框架**：1分钟 / 5分钟 / 15分钟 / 60分钟 / 日线 / 周线 / 月线 / 季线
- **10 种滤波算法**：Savgol、EMA、Kalman、Butterworth 等
- **双滤波器并行**：主滤波器 + 对比滤波器同时运行
- **Schmitt Trigger 自适应信号**：滞回状态机 + 波动率自适应死区
- **抛物线预测 + PnL 回测**：物理抛物线拟合 + 两阶段退出策略
- **跨周期 PnL 参考**：紧邻高周期交易事件同步到本周期子图

```

  │
  ├─[梯度]──→ v = d(filtered)/dt

  │
  ├─[Schmitt Trigger]──→ sig_t (+1/0/-1)
  │   ├─ epsilon_t = k_eps × max(σ_t(v), σ_min)
  │   └─ 滞回状态机

  │
  ├─[_fit_physics_parabola]──→ 预测曲线
  │   └─ y = a(x-x₀)² + y₀ (锚定对尾)
  │

      └─ 两阶段退出 (保护期 + 趋势跟踪)
```

#### 2.3.1 跨子图十字准线

系统通过自定义 Plotly JS 实现了**跨所有子图的统一十字准线**——鼠标在一个子图上移动时，所有同级子图同步显示垂直参考线。

**实现方式（Plotly post_script + JavaScript）**：

```
Plotly Figure
  │
  └─ post_script (JavaScript 注入)
       │
       ├─ 监听 'plotly_hover' 事件
       │    └─ 捕获 hover 点的 x 坐标
       │
       ├─ 计算相对 x 位置 (0~1, 归一化到 paper 坐标)
       │
       └─ 在所有子图上绘制/移动垂直 line shape
            ├─ 线条样式: 虚线 (dash: 'dot')
            ├─ 颜色: 灰色半透明 rgba(128,128,128,0.4)
            └─ 层级: layer 'above' (在所有数据之上)
```

**关键设计决策**：
- **自定义 JS 而非 Plotly 内置 spikeline**：Plotly 内置的 `spikemode` 仅在同一 `xaxis` 上有效，无法跨 `xaxis`（如 Row 6/7/8 有独立 x 轴但共享时间对齐）
- **post_script 注入**：通过 `fig._config['post_script']` 在图表渲染后在浏览器端执行 JS，不增加 Python 端计算开销
- **性能**：十字准线在 `plotly_hover` 事件中实时更新，无额外数据查询

**行为细节**：
- hover 在价格子图（Row 1）→ 所有子图显示竖线
- hover 在 PnL 子图（Row 6/7/8）→ 同上，竖线与价格子图对齐
- 鼠标移出图表区域 → 竖线消失（`plotly_unhover` 事件触发清理）
- subplot 条件性显示（如 Row 7/8 未启用）不影响现有子图的十字准线功能

### 2.4 4 视图渲染顺序与 PnL 数据流

系统按**固定顺序**渲染 4 个视图，顺序直接影响跨周期 PnL 数据的可用性。

#### 2.4.1 渲染顺序

```
高周期优先渲染 → 低周期后续渲染

v3 (周线) → v1 (日线) → v2 (60分钟) → v0 (15分钟)
  │             │            │             │
  └── 写入 ──→ session_state  └── 写入 ──→ session_state
       ↑                              ↑
       └──── 读取 ──── v1/v2 ────────┘
```

**关键规则**：
- 高周期视图先计算完 PnL → 写入 `session_state[f'pnl_data_{tf}']`
- 低周期视图渲染时读取已写入的高周期 PnL → 用于跨周期子图
- 若高周期视图因数据不足未能渲染 → 低周期视图的跨周期子图为空

#### 2.4.2 PnL 数据流（逐级写入/读取）

```
_render_chart(v3, "周线")
  │
  ├─ compute schmitt → predict → PnL
  ├─ st.session_state['pnl_data_周线'] = {long_pnl, short_pnl, trades}
  │
  ▼
_render_chart(v1, "日线")
  │
  ├─ compute schmitt → predict → PnL
  ├─ 读取 st.session_state['pnl_data_周线'] → 跨周期参考子图数据源
  ├─ st.session_state['pnl_data_日线'] = {long_pnl, short_pnl, trades}
  │
  ▼
_render_chart(v2, "60分钟")
  │
  ├─ ...读取 pnl_data_日线...
  ├─ st.session_state['pnl_data_60分钟'] = ...
  │
  ▼
_render_chart(v0, "15分钟")
  │
  ├─ ...读取 pnl_data_60分钟...
  └─ st.session_state['pnl_data_15分钟'] = ...
```

#### 2.4.3 跨周期数据获取规则

| 本周期 | 查找高周期 | 查找策略 |
|--------|-----------|---------|
| 1分钟 | 5分钟 | 紧邻下一个周期 |
| 5分钟 | 15分钟 | 紧邻下一个周期 |
| 15分钟 | 60分钟 | 紧邻下一个周期 |
| 60分钟 | 日线 | 紧邻下一个周期 |
| 日线 | 周线 | 紧邻下一个周期 |
| 周线 | 月线 | 紧邻下一个周期 |
| 月线 | 季线 | 紧邻下一个周期 |
| 季线 | 无 | 不显示跨周期子图 |

**无跨周期跳级查找**：15 分钟不会直接查找日线的 PnL（跳过了 60 分钟）。这一限制意味着若中间周期视图未开启策略，跨周期子图将为空——即使更高级别有可用数据。

#### 3.2.1 `_imp` 参数备份机制

为支持参数重置功能，每个可调参数键在 JSON 配置中同时保留一个 `_imp` 后缀的备份键：

| 参数键 | 备份键 | 作用 |
|--------|--------|------|
| `v0_ke` | `v0_ke_imp` | 保存原始导入值 |
| `v0_sm` | `v0_sm_imp` | 保存原始导入值 |
| `v0_ew` | `v0_ew_imp` | 保存原始导入值 |
| … (所有可调参数) | `_imp` 后缀 | 保存原始导入值 |

**工作流程**：
1. **导入时**：JSON 中 `_imp` 键的值被读入内存作为"原始值"
2. **UI 滑块修改时**：仅修改运行时参数（如 `v0_ke`），不改变 `v0_ke_imp`
3. **点击"重置"时**：将 `v0_ke` 恢复为 `v0_ke_imp` 的值
4. **导出配置时**：用户可选择"导出当前值"或"导出原始值"

**设计意图**：`_imp` 机制实现了配置的"出厂设置"保留——用户可随意调整参数，始终可用一键重置回到导入时的状态，无需手动记忆或重新导入配置文件。

### 3.4 参数依赖链

配置参数之间存在**严格逐级启用依赖**。若上游开关关闭，下游开关即使 UI 勾选也**无实际效果**。

```
sch (Schmitt Trigger)
  └─→ pred (抛物线预测)          ← sch 关闭时 pred 不执行
       └─→ strat (策略 PnL)      ← pred 关闭时 strat 不执行
            └─→ cross_pnl (跨周期) ← strat 关闭时 cross_pnl 不执行
                 └─→ align (同向性) ← cross_pnl 关闭时 align 不执行
```

**依赖验证规则**：

| 下游参数 | 上游前提 | 违反时的行为 |
|---------|---------|------------|
| `v{N}_pred` | `v{N}_sch = true` | pred 开关在 UI 置灰（不可操作） |
| `v{N}_strat` | `v{N}_pred = true` | strat 开关在 UI 置灰 |
| `v{N}_cross_pnl` | `v{N}_strat = true` | cross_pnl 开关在 UI 置灰 |
| `v{N}_align` | `v{N}_cross_pnl = true` | align 开关在 UI 置灰 |

**常见误区**：
- **pred=true 但 sch=false**：抛物线拟合需要 Schmitt 信号以生成多空对，无信号时无 pair 可拟合，预测线不显示
- **strat=true 但 pred=false**：策略 PnL 依赖预测方向确认入场，无预测时所有信号被跳过（收不到入场确认）
- **cross_pnl=true 但高周期视图未开 strat**：跨周期子图显示但无数据源，表现为空白行

**设计意图**：依赖链确保信号链路的完整性——每层都依赖上游产出合法的中间结果。逐级灰化避免用户误配置。

### 3.5 滤波参数键命名规则

滤波参数键遵循严格的命名约定：**`{中文标签}_{视图}_f{滤波器编号}_{算法ID}`**

| 组成部分 | 含义 | 示例 |

以下约束在代码中**硬编码**，不体现在配置文件或 UI 中，用户需知晓以避免预期外的行为。

| # | 约束 | 影响 | 位置 |

| 8 | **n_extend 影响 PnL 计算** | 预测延伸数 = 保护期长度；n_extend 越大，止损保护期越长，但也引入更多 look-ahead | `filter_app/streamlit_app.py` |
| 9 | **同一 ticker 跨视图必需** | 4 个视图**必须**使用同一 ticker；切换标的会清空所有视图的 PnL 缓存 | `filter_app/streamlit_app.py` |

- 约束 1、3、9 违反后可能导致空白图表或 Python 异常
- 约束 6 违反后功能静默失效（无错误提示）

## 4. 滤波算法

### 4.1 已注册滤波器（10 种）

| 算法 | 函数 | 参数 | 特点 |
|------|------|------|------|
| **Savgol** | `savgol_filter` | window, order | 多项式局部拟合，保趋势形状 |
| **EMA** | 指数加权 | span | 自适应平滑，近期权重高 |
| **SMA** | 简单移动平均 | window | 等权平滑，滞后明显 |
| **WMA** | 加权移动平均 | window | 线性递减权重 |
| **ALMA** | Arnaud Legoux | window, offset, sigma | 低滞后高斯滤波 |
| **Kalman** | 1D 恒速卡尔曼 | Q, R | 贝叶斯最优估计 |
| **Butterworth** | IIR 低通 | order, cutoff | 频域设计，陡峭截止 |
| **Gaussian** | 高斯滤波 | sigma | 对称权重平滑 |
| **Median** | 中值滤波 | window | 抗异常值 |
| **LOWESS** | 局部加权回归 | frac | 非参数鲁棒拟合 |

### 4.2 Dual Filter 模式

当 `global_dual=true` 时：
- 主滤波器（如 Savgol）→ 生成 `filtered`，用于 Schmitt Trigger + PnL 计算
- 第二滤波器（如 EMA）→ 生成 `filtered2`，仅在价格子图中作为视觉对比线
- **注意**: PnL 仅使用主滤波器结果，第二滤波器不参与交易决策

## 5. Schmitt Trigger 信号生成

### 5.1 核心公式

$$
\varepsilon_t = k_{eps} \cdot \max(\sigma_t(v), \sigma_{min})
$$

其中：
- $\sigma_t(v)$ = 速度 v 的 EWMA 波动率（$\alpha = 2/(N_{EWMA}+1)$）
- $k_{eps}$ = 灵敏度系数（配置键 `ke`）
- $\sigma_{min}$ = 地板保护（配置键 `sm`）

### 5.2 状态机（非对称滞回）

```
          a > +ε AND v > 0
    0 ───────────────────────→ +1
                                  │
                          a < -ε  │
                                  ▼
    0 ←─────────────────────── +1

          a < -ε AND v < 0
    0 ───────────────────────→ -1
                                  │
                          a > +ε  │
                                  ▼
    0 ←─────────────────────── -1
```

**入场需双条件**（a + v），**离场仅需单条件**（a）。非对称设计天然抑制噪音翻转。

### 5.3 参数调优

| 参数 | 降低效果 | 升高效果 |
|------|---------|---------|
| `ke` (k_eps) | 更窄死区，更多信号 | 更宽死区，更少信号 |
| `sm` (sigma_min) | 低波动保护更弱 | 低波动保护更强 |
| `ew` (N_EWMA) | 波动率自适应更快 | 波动率估计更平滑 |

## 6. 抛物线预测

### 6.1 两种拟合模式

**Physics Parabola**（`fm="parabola"`）:
$$y = a(x - x_0)^2 + y_0$$

- 锚定对尾为顶点：$(x_0, y_0) = (x_{end}, y_{end})$
- 仅拟合曲率 $a$（1 自由度）
- 物理含义：价格从当前顶点按抛物线轨迹发散

**Standard Poly2**（`fm="poly2"`）:
$$y = ax^2 + bx + c$$

- 标准二次多项式（3 自由度）
- 无约束最小二乘

#### 6.1.1 拟合失败条件

抛物线拟合在以下两类条件下判定失败，失败后静默跳过该多空对（不报错、不中断渲染）：

| 失败条件 | 判定逻辑 | 适用模式 | 说明 |
|---------|---------|---------|------|
| **数据点不足** | `len(pair_prices) < 3` | parabola / poly2 | 抛物线拟合最少需要 3 个数据点；多空对区间过短时无法拟合 |
| **分母趋零** | `|denominator| < 1e-12` | 仅 parabola | 曲率参数 $a = \text{numerator} / \text{denominator}$，分母趋零意味着数据近似线性，曲率无定义 |

**失败处理流程**：
```
_find_all_pairs() → 生成多空对列表
    │
    ▼
对每个 pair:
    _fit_physics_parabola(pair)
        │
        ├─ 成功 → 生成预测曲线 → 传入 _compute_strategy_pnl()
        │
        └─ 失败 → continue（跳过该 pair，不生成交易）
```

**设计意图**：拟合失败是正常现象（如极短的信号区间、价格近乎直线的盘整期），不应视为错误。静默跳过保持了 PnL 计算的连续性，其余合法的多空对不受影响。

### 6.2 预测延伸

对每个多空对（pair），拟合完成后向未来延伸 `n_extend`（默认 8）个 bar：

$$x_{pred} \in [x_{end}+1, x_{end}+n_{extend}]$$

预测方向 = 预测末点 vs 预测起点的价格比较。

### 6.3 fit_mode 行为差异详解

#### 6.3.1 两种模式的根本差异

| 维度 | Physics Parabola (`fm="parabola"`) | Standard Poly2 (`fm="poly2"`) |
|------|-----------------------------------|-------------------------------|
| 公式 | `y = a(x-x₀)² + y₀` | `y = ax² + bx + c` |
| 自由度 | 1（仅曲率 a） | 3（a, b, c） |
| 锚点约束 | 顶点 = pair 尾点（强制） | 无约束 |
| 物理含义 | 价格从当前顶点按抛物线发散 | 纯数学最小二乘拟合 |
| 对噪声的鲁棒性 | 高（自由度低） | 低（自由度高易过拟合） |
| 趋势转折响应 | 慢（顶点锚定限制方向变化） | 快（可自由调整方向） |
| 推荐场景 | 趋势市中继预测 | 震荡市转折点捕捉 |

#### 6.3.2 模式选择决策树

```
是否期望价格延续当前趋势？
  ├─ 是 → 趋势市 → parabola（锚定顶点，保守预测）
  └─ 否 → 震荡市/转折点 → poly2（自由拟合，激进预测）

数据噪声水平？
  ├─ 低噪声（日线/周线）→ parabola 或 poly2 均可
  └─ 高噪声（分钟级）  → parabola（低自由度抑制过拟合）
```

#### 6.3.3 实际效果对比

| 场景 | parabola 表现 | poly2 表现 |
|------|--------------|-----------|
| 单边上涨趋势 | 预测延续上涨，精度高 | 可能过度拟合短期回调，预测偏保守 |
| 横盘震荡 | 预测近似水平线（曲率→0） | 拟合局部震荡，预测方向频繁切换 |
| V 形反转 | 预测滞后（锚定在旧顶点） | 更快捕捉转折（3 自由度可重构方向） |
| 数据点极少（<5） | 分母趋零风险高 → 拟合可能失败 | 3 点即可拟合（但结果不可靠） |

## 7. 策略 PnL 计算

### 7.1 入场条件

对于每个多空对的结束点（`pair_end`）：

| 信号 | 条件 |
|------|------|
| **做多** | `sig_t[pair_end] == +1` **且** 预测方向向上 |
| **做空** | `sig_t[pair_end] == -1` **且** 预测方向向下 |

不满足条件的信号对 → 跳过。

### 7.2 两阶段退出（Scheme D）

```
阶段1: 预测保护期 [entry+1, entry+N_ext]
  ├─ 止损: |实际价格 - 预测价格| > stop_loss_pct%
  └─ 止盈: Schmitt 信号反转

阶段2: 趋势跟踪期 [entry+N_ext+1, ...]
  └─ 止盈: 仅 Schmitt 信号反转（让利润奔跑）
```

#### 7.2.1 离场原因完整列表

系统支持三种离场类型，每种有明确的触发条件和标记行为：

| 离场类型 | 代码标识 | 触发条件 | 发生阶段 | 标记符号 | 标注颜色 |
|---------|---------|---------|---------|---------|---------|
| **止盈** | `take_profit` | Schmitt 信号反转（`sig_t` 从 ±1 变为 0 或反向） | 阶段1（保护期）+ 阶段2（趋势跟踪） | ○ circle | 绿色 `#3fb950` |
| **止损** | `stop_loss` | `|实际价格 - 预测价格| / 预测价格 > stop_loss_pct%` | 仅阶段1（保护期） | ✕ x | 红色 `#f85149` |
| **日终强制平仓** | `eod` | 到达最后一个 bar（`i == len(data)-1`）时仍有持仓 | N/A（数据边界） | **不画标记** | N/A |

**eod 特殊处理**：
- eod 离场**不在图表上绘制任何标记**（无 circle / x 符号，无文字标注）
- eod 离场的收益率正常计入 PnL 统计（交易笔数、胜率、总收益）
- 设计理由：eod 是数据边界而非策略信号，标记它会产生误导（暗示策略在此时做出了离场决策）
- 用户可通过交易列表（hover 信息）查看 eod 离场的具体数值

#### 7.2.2 阶段判定流程图

```
入场 pair_end
  │
  ▼
bar i in [entry+1, entry+N_ext]?  ──Yes──→ 阶段1: 止损检查 + 止盈检查
  │                                              │
  No                                         任一触发 → 离场
  │
  ▼
bar i in [entry+N_ext+1, ...]?   ──Yes──→ 阶段2: 仅止盈检查
  │                                              │
  No                                         触发 → 离场
  │
  ▼
i == len(data)-1?  ──Yes──→ eod 强制平仓（不画标记）
```

### 7.3 PnL 计算

- **初始资金**: 100.0（做多和做空各自独立）
- **做多收益**: `(exit_price - entry_price) / entry_price`
- **做空收益**: `(entry_price - exit_price) / entry_price`
- **非持仓期**: 水平直线（前向填充最后已知资金值）

#### 7.3.1 资金管理模式

做多和做空各自维护**独立资金池**，互不干扰：

| 属性 | 做多资金池 | 做空资金池 |
|------|----------|----------|
| 初始资金 | 100.0 | 100.0 |
| 累积方式 | 连续复利累积 | 连续复利累积 |
| 更新时机 | 仅做多交易离场时 | 仅做空交易离场时 |
| 非持仓期 | 水平直线（`capital[t] = capital[t-1]`） | 水平直线 |

**累积公式**：

$$\text{capital}_{t} = \text{capital}_{t-1} \times (1 + r_t)$$

其中 $r_t$ 为单笔交易收益率（做多: `(exit - entry) / entry`，做空: `(entry - exit) / entry`）。

**示例**：假设做多资金池经历三笔交易：
```
初始: 100.0
第1笔: +5%  → 100.0 × 1.05 = 105.0
第2笔: -2%  → 105.0 × 0.98 = 102.9
第3笔: +3%  → 102.9 × 1.03 = 105.987
```
交易间隔期间 PnL 曲线为水平直线（前向填充），仅在离场时刻发生阶跃变化。

**设计意图**：独立资金池模拟了做多策略和做空策略各自独立运行的真实场景——做多的浮盈不用于做空保证金，做空的浮盈不用于做多开仓。用户可分别评估多头和空头策略的独立表现。

### 7.4 统计指标

| 指标 | 说明 |
|------|------|
| 交易笔数 | 总交易次数 |
| 胜率 | 正收益交易 / 总交易 |
| 做多收益 | long_pnl[-1] - 100.0 |
| 做空收益 | short_pnl[-1] - 100.0 |
| 最大回撤 | (PnL - 峰值) / 峰值 × 100 |

### 7.5 信号重绘修复方案

#### 7.5.1 问题定义

当前回测中抛物线拟合存在 **look-ahead bias**（已在 §13 限制中标注）。具体路径：

```
pair 区间: [pair_start, pair_end]
拟合数据: pair_start 到 pair_end 的所有价格点（含 pair_end 之后的 bar）
                                        ↑
                                   拟合时"看到"了未来数据
```

**偏差量级估算**：

| 周期 | n_extend=8 时偏差 | n_extend=15 时偏差 |
|------|-------------------|---------------------|
| 15分钟 | ~2 小时 | ~3.75 小时 |
| 60分钟 | ~8 小时 | ~15 小时 |
| 日线 | ~8 天 | ~15 天 |
| 周线 | ~8 周 | ~15 周 |

#### 7.5.2 修复方案

**方案 A：严格因果拟合（推荐用于实盘）**

拟合抛物线时仅使用 `[pair_start, pair_end]` 的数据点，不包含任何未来 bar。拟合完成后向未来延伸 n_extend 个 bar 仅用于**预测显示**，不参与曲率参数计算。

```
修复后：
  拟合数据: pair_start 到 pair_end（不含未来点）
  预测显示: pair_end+1 到 pair_end+n_extend（仅可视化）
              ↑
         预测不反哺拟合参数
```

**影响**：
- 拟合精度略降（少用 n_extend 个数据点），但消除前视偏差
- 回测 PnL 将比当前值**保守**（当前值因 look-ahead 而偏高）
- Schmitt 信号生成不受影响（信号仅依赖滤波价格一阶/二阶导）

**方案 B：双模式开关**

添加配置键 `v{N}_causal_fit`（默认 `false` 保持向后兼容），设为 `true` 时启用方案 A。

```
if causal_fit:
    fit_data = prices[pair_start : pair_end+1]      # 仅用已知数据
else:
    fit_data = prices[pair_start : pair_end+n_extend] # 当前行为（含未来）
```

#### 7.5.3 实施优先级

| 阶段 | 措施 | 影响 |
|------|------|------|
| 立即 | 在 §13 限制中明确标注量级（已完成） | 用户知情 |
| 短期 | 实现方案 B（双模式开关） | 可对比因果 vs 非因果回测 |
| 中期 | 实盘管线默认使用 causal_fit=true | 消除前视偏差 |

### 7.6 ATR 动态止损

#### 7.6.1 固定止损的局限

当前系统仅支持固定百分比止损（§7.2 中的 `stop_loss_pct%`）。问题是：
- 高波动期固定止损过早触发（正常波动被误判为止损）
- 低波动期固定止损过晚触发（实际亏损远超预期）

#### 7.6.2 ATR 动态止损方案

**ATR (Average True Range)** 自适应调整止损距离：

```
动态止损距离 = max(stop_loss_pct%, N × ATR(14) / price)
```

| 市场状态 | ATR(14) | 固定 2% 止损 | 动态止损 (N=2) | 效果 |
|---------|---------|-------------|---------------|------|
| 低波动 | 0.5% | 2% | max(2%, 1.0%) = 2% | 与固定相同 |
| 正常 | 1.5% | 2% | max(2%, 3.0%) = 3% | 放宽止损，减少误杀 |
| 高波动 | 3.0% | 2% | max(2%, 6.0%) = 6% | 大幅放宽，避免波动止损 |

#### 7.6.3 实现要点

```python
def compute_dynamic_stop(entry_price, atr_14, stop_loss_pct, atr_multiplier=2.0):
    """
    计算动态止损价格
    entry_price: 入场价
    atr_14: 当前 bar 的 14 周期 ATR 值
    stop_loss_pct: 配置的最小止损百分比（如 0.02）
    atr_multiplier: ATR 倍数（建议 1.5-2.5）
    """
    fixed_stop = entry_price * stop_loss_pct
    atr_stop = atr_multiplier * atr_14
    stop_distance = max(fixed_stop, atr_stop)
    return entry_price - stop_distance  # 做多; 做空取 +
```

#### 7.6.4 参数建议

| 时间框架 | atr_multiplier | 说明 |
|---------|---------------|------|
| 15分钟 | 1.5 | 噪声高，较小倍数减少误杀 |
| 60分钟 | 2.0 | 平衡 |
| 日线 | 2.5 | 趋势明确，较大倍数让利润奔跑 |
| 周线 | 3.0 | 长线持有，容忍更大回撤 |

- **双重编码**：颜色（离场原因）+ 箭头（多空方向）各自独立，无信息过载
- **视觉一致**：三行标注共享同一套规则，降低认知负担
- **零额外空间**：箭头仅 +1 字符，不挤占图表空间
- **与标记联动**：标注颜色 = 标记颜色，视觉上形成完整的信息单元

## 9. 同向性判断子图

### 9.1 功能说明

在本周期 PnL 子图（row 6）和高周期参考子图（row 7）下方，新增 **Row 8 同向性判断子图**。

**核心逻辑**：高周期处于做多/做空持仓区间时，采样本周期同方向 PnL 涨跌幅；高周期非持仓时，维持上一个值不变（水平线）。

```
对于本周期每个 bar i：
  若 高周期做多持仓区间 → pnl[i] = pnl[i-1] × (Row6做多PnL[i] / Row6做多PnL[i-1])
  若 高周期做空持仓区间 → pnl[i] = pnl[i-1] × (Row6做空PnL[i] / Row6做空PnL[i-1])
  否则               → pnl[i] = pnl[i-1]  （hold，水平直线）
```

### 11.5 三阶段参数调优方法论

系统性参数调优分为三个阶段，逐级收敛最优配置。

#### 阶段 1: 粗调（确定数量级）

**目标**: 找到参数的有效范围，排除明显不合理的配置。

| 步骤 | 操作 | 验证标准 |
|------|------|---------|
| 1.1 | 固定 ke=0.15, sm=0.05，扫描 Savgol window=[5, 51]（步长 4） | 信号数量在合理范围（每 100 bar 有 3-15 次切换） |
| 1.2 | 固定阶段 1.1 最优 window，扫描 ke=[0.05, 0.50]（步长 0.05） | 死区宽度不会吃掉所有信号或放过所有噪声 |
| 1.3 | 固定前两步最优值，扫描 n_extend=[3, 30]（步长 3） | 预测保护期覆盖大部分正常回撤 |
| 1.4 | 固定前三步，扫描 stop_loss_pct=[0.5%, 10%]（步长 0.5%） | 止损不频繁触发（胜率 > 30%） |

**输出**: 参数大致范围，排除极端值。

#### 阶段 2: 精调（网格搜索 + Walk-Forward）

**目标**: 在阶段 1 范围内精细搜索，使用 Walk-Forward 避免过拟合。

| 步骤 | 操作 | 验证标准 |
|------|------|---------|
| 2.1 | 网格搜索：(ke, sm, ew, window, n_extend, sl) 六维网格 | 训练集/验证集 PnL 相关性 > 0.6 |
| 2.2 | Walk-Forward: 训练 60% → 验证 20% → 测试 20% | 三期 PnL 均为正 |
| 2.3 | 敏感性分析：每个参数 ±20%，观察 PnL 变化 | 参数稳定区（非悬崖边缘） |

**输出**: 一组经过时间鲁棒性验证的参数。

#### 阶段 3: 验证（多标的 + 多周期）

**目标**: 确认参数非针对单标的过度优化。

| 步骤 | 操作 | 验证标准 |
|------|------|---------|
| 3.1 | 同参数应用到 2-3 个同类标的（如同属恒生科技） | PnL 方向一致（至少不出现系统性亏损） |
| 3.2 | 同参数应用到不同时间框架 | 高周期应更稳定（交易更少但胜率更高） |
| 3.3 | 样本外测试（最近 3 个月数据，调优时未使用） | PnL 与样本内偏差 < 30% |

#### 调优陷阱警示

| 陷阱 | 表现 | 避免方法 |
|------|------|---------|
| **过拟合** | 训练集 PnL 极佳，样本外崩溃 | Walk-Forward 验证；限制参数搜索范围 |
| **幸存者偏差** | 仅看最终 PnL，忽略中间回撤 | 检查最大回撤和回撤持续时间 |
| **数据窥探** | 用全量数据调参后再"测试" | 严格时间分割（先划分再调参） |
| **忽略交易成本** | 高分频策略 PnL 虚高 | 加入滑点 + 佣金估算（每笔 0.1-0.3%） |

## 12. 数据层

### 12.1 SQLite 表结构

```sql
CREATE TABLE kline (
    ticker TEXT,
    timeframe TEXT,
    ts TEXT,          -- 时间戳
    open REAL, high REAL, low REAL, close REAL, volume REAL,
    PRIMARY KEY (ticker, timeframe, ts)
);
```

#### 12.1.1 Upsert 分裂策略

为避免单条 SQL 语句过长导致的性能和事务问题，`upsert_kline()` 采用**分批插入**策略：

| 参数 | 值 | 说明 |
|------|---|------|
| 批次大小 | 500 行/批 | 每批 INSERT 的行数上限 |
| 冲突策略 | `INSERT OR REPLACE` | 主键冲突时覆盖整行（同 ticker+timeframe+ts） |
| 分批方式 | `range(0, len(df), batch_size)` | 按 DataFrame 索引切片 |
| 事务边界 | 每批一个隐式事务 | SQLite 单条语句自动事务 |

```
DataFrame (N 行)
  │
  ├─ batch 0: rows[0:500]    → INSERT OR REPLACE
  ├─ batch 1: rows[500:1000] → INSERT OR REPLACE
  ├─ batch 2: rows[1000:1500] → INSERT OR REPLACE
  └─ ...
```

**设计理由**：`INSERT OR REPLACE` 在唯一约束冲突时先 DELETE 再 INSERT，确保增量更新不产生重复行，同时保持写入幂等（重复调用不改变最终状态）。

#### 12.1.2 SQLite 连接配置

系统在 `db.py` 中配置 SQLite 连接参数，优化并发读写场景下的性能和可靠性：

| 参数 | 值 | 说明 |
|------|---|------|
| `journal_mode` | `WAL` | Write-Ahead Logging，支持并发读写：写操作不阻塞读操作 |
| `synchronous` | `NORMAL` | 平衡安全性与性能：WAL 模式下 crash-safe，比 FULL 快 2-3x |
| `busy_timeout` | `5000` (ms) | 锁等待超时：并发写入时等待最多 5 秒后报错 |
| `row_factory` | `sqlite3.Row` | 查询结果以字典式 Row 对象返回，支持 `row['close']` 按列名访问 |

**连接管理**：
- 每次请求通过 `get_db()` 获取连接，使用 `with` 上下文管理器自动关闭
- 不在线程间共享连接对象（每个线程独立连接）
- 连接在 Streamlit 会话生命周期内复用（`@st.cache_resource`）

**WAL 模式优势**：8 个并行下载线程可同时写入 SQLite 而不会触发 `database is locked` 错误。

#### 12.1.3 数据层辅助函数

`db.py` 提供以下辅助函数，供 Streamlit 主程序调用：

| 函数 | 签名 | 功能 | 返回值 |
|------|------|------|--------|
| `has_data(ticker, tf)` | `(str, str) → bool` | 检查指定标的+周期是否有数据 | `True`/`False` |
| `get_date_range(ticker, tf)` | `(str, str) → (str, str)` | 获取数据的起止日期范围 | `(min_ts, max_ts)` 或 `None` |
| `init_db()` | `() → None` | 初始化数据库：创建 `kline` 表（如不存在），设置 WAL 模式 | 无 |
| `upsert_kline(ticker, tf, df)` | `(str, str, DataFrame) → int` | 分批 upsert K 线数据 | 写入总行数 |
| `load_kline(ticker, tf)` | `(str, str) → DataFrame` | 按时间排序加载 K 线数据 | DataFrame（可能为空） |
| `get_available_timeframes(ticker)` | `(str) → list[str]` | 获取指定标的所有已有数据的时间框架列表 | 如 `['15分钟', '日线', '周线']` |

所有函数内部捕获异常，失败时返回安全默认值（空 DataFrame / `False` / `None`），不向上抛出异常。

#### 12.2.1 yfinance Period 动态选择

系统根据所需数据点数和时间框架，自动选择最优的 `yfinance` period 参数，确保获取足够的历史数据覆盖回看窗口。

**日线/周线/月线 — 7 级阶梯式 Period 选择：**

| 条件 (n_pts) | period | 安全裕度 | 说明 |
|-------------|--------|---------|------|
| n_pts ≤ 5 | `1mo` | ×1.5 | 最小覆盖，轻量请求 |
| n_pts ≤ 20 | `3mo` | ×1.5 | |
| n_pts ≤ 60 | `6mo` | ×1.5 | |
| n_pts ≤ 120 | `1y` | ×2 | 默认配置常用档位 |
| n_pts ≤ 250 | `2y` | ×2 | |
| n_pts ≤ 500 | `5y` | ×2 | |
| n_pts > 500 | `max` | ×5 | 最大历史范围 |

选择逻辑：`n_pts × 安全裕度` 确保数据点数远超回看窗口，为滤波器和信号计算提供足够的预热数据。

**分钟级 — 强制 Period 映射：**

| 时间框架 | period | 说明 |
|---------|--------|------|
| 1分钟 | `7d` | yfinance 分钟数据最多提供 7 天历史 |
| 5分钟 | `1mo` | 5分钟数据最多提供 60 天历史 |
| 15分钟 | `1mo` | |
| 60分钟 | `2y` | 60分钟数据最多提供 730 天历史 |

分钟级数据的 period 由 yfinance API 上限强制约束，系统在 API 限制内取最大可用范围。

#### 12.2.2 并行获取架构

系统使用 `ThreadPoolExecutor`（8 线程）并行获取所有时间框架的 K 线数据：

```
主线程
  │
  ├─[ThreadPoolExecutor(max_workers=8)]
  │   │
  │   ├─ Thread 1: fetch("1分钟")
  │   ├─ Thread 2: fetch("5分钟")
  │   ├─ Thread 3: fetch("15分钟")
  │   ├─ Thread 4: fetch("60分钟")
  │   ├─ Thread 5: fetch("日线")
  │   ├─ Thread 6: fetch("周线")
  │   ├─ Thread 7: fetch("月线")
  │   └─ Thread 8: fetch("季线")
  │
  └─[结果汇总]
      ├─ 单个线程异常不影响其他线程
      ├─ 每个线程独立 upsert 到 SQLite（WAL 模式支持并发写）
      └─ 汇总结果返回给 Streamlit 缓存层
```

**容错设计**：
- 单个时间框架获取失败 → 该框架返回空 DataFrame，不阻塞其他框架
- 全部框架获取失败 → UI 显示错误信息，已缓存的旧数据仍可渲染
- 线程超时由 `yfinance` 内部超时控制，不额外设置全局超时

#### 12.2.3 Parquet 显示缓存链路

为了加速 Streamlit 图表渲染，系统在 SQLite 之上增加了一层 Parquet 缓存：

```
yfinance API
    │
    ▼
SQLite (market.db) ─── upsert_kline()   ← 持久层
    │
    ▼
_sync_to_display() ─── 筛选 ticker + timeframe
    │                    按 ts 排序
    ▼
Parquet 文件 ─── data/display/{ticker}/{tf}.parquet   ← 显示缓存
    │
    ▼
pd.read_parquet() ─── 极速读取（比 SQLite 查询快 5-10x）
    │
    ▼
_render_chart() ─── 传入 Plotly
```

**缓存刷新策略**：
- 每次点击"刷新数据"时，先更新 SQLite，再同步到 Parquet
- Parquet 文件按 `{ticker}/{timeframe}` 组织，支持单框架增量更新
- 若 Parquet 文件不存在或损坏 → 自动从 SQLite 重建，对用户透明

#### 12.2.4 Streamlit 缓存配置

系统使用 `@st.cache_data` 装饰器管理数据缓存，减少重复计算和 API 调用：

| 缓存函数 | TTL (秒) | 说明 |
|---------|---------|------|
| `load_config()` | 3600 (1h) | 配置文件解析结果 |
| `get_kline_data()` | 600 (10min) | 从 Parquet/SQLite 读取的 K 线数据 |
| `fetch_all_timeframes()` | 300 (5min) | yfinance API 批量获取结果 |
| `compute_filtered_data()` | 无 TTL (hash-based) | 基于输入参数哈希自动失效 |

- **hash-based 缓存**：滤波计算结果以 `(data_hash, filter_params_hash, schmitt_params_hash)` 为键，参数不变时直接命中缓存
- **手动刷新**：点击"刷新数据"按钮清除所有 TTL 缓存，强制重新获取
- **自动刷新**：可配置间隔（默认关闭），后台定时清除数据获取缓存

### 12.3 时间窗口导航

- `day_offset`: 全局偏移天数，4 视图共享
- 步长: 1/3/5/10/20/30/60/90/180/365 天
- 前移: 查看更早历史数据
- 后移/最新: 回到近期数据

### 12.4 异常处理策略

系统在数据层的异常处理遵循"静默降级、不阻断渲染"的原则。

| 场景 | 策略 | 行为 |
|------|------|------|
| **upsert 写入失败** | 静默捕获 | 记录警告日志，不抛出异常，当前会话继续使用已有缓存数据 |
| **滤波计算异常** | fallback 到原始值 | 若滤波器对某段数据计算失败（如窗口不足），`filtered = raw`，梯度/加速度置零，该区间不产生交易信号 |
| **yfinance API 错误** | 展示错误信息 + 降级 | 网络超时/限流/无数据时，在 UI 中显示错误提示，已缓存的数据正常渲染；新数据获取失败不阻塞图表显示 |
| **Parquet 缓存读写失败** | 静默降级 | 缓存写入失败时跳过缓存，直接从 SQLite 读取；缓存读取失败时回退到 SQLite 查询 |
| **Schmitt Trigger 计算失败** | 信号置零 | 异常区间 `sig_t = 0`，不产生多空对，PnL 曲线在该区间保持水平 |
| **抛物线拟合失败** | 跳过该对 | 详见 §6.1 拟合失败条件，失败后静默跳过该多空对，不报错 |

#### 12.5.4 P0 边界条件审计

对 25 项边界条件进行审计，以下为 P0 级别发现：

| # | 边界条件 | 严重性 | 现象 | 根本原因 |
|---|---------|--------|------|---------|
| 1 | **配置导入无校验** | 高 | 非法 JSON 或缺失键导致运行时 KeyError | `load_config()` 无 schema 验证 |
| 2 | **切换标的未重置 offset** | 中 | 新标的数据窗口错位，显示空白图表 | `day_offset` 未随 ticker 变化重置 |
| 3 | **compact 参数硬编码** | 低 | 约 340px 垂直空间的死代码 | `compact=True` 硬编码，`compact=False` 分支不可达 |
| 4 | **同周期 PnL 覆盖** | 中 | 后渲染的同周期视图覆盖先渲染的 PnL 缓存键 | `session_state` 键以 `tf` 而非 `view_id` 命名 |
| 5 | **无跨周期跳级查找** | 中 | 中间周期缺失时跨周期子图为空 | 查找逻辑仅检查紧邻周期（见 §2.4.3） |

#### 12.5.5 配置安全建议

```python
# 当前：无验证
config = json.load(f)  # 任意 JSON 均被接受

# 建议：schema 验证
REQUIRED_GLOBAL_KEYS = ['market', 'ticker', 'global_f', 'global_f2', 'global_dual']
REQUIRED_VIEW_KEYS = ['tf', 'n', 'sch', 'pred', 'strat', 'ke', 'sm', 'ew', ...]

def validate_config(config: dict) -> list[str]:
    errors = []
    for key in REQUIRED_GLOBAL_KEYS:
        if key not in config:
            errors.append(f"缺少全局键: {key}")
    for v in ['v0', 'v1', 'v2', 'v3']:
        for key in REQUIRED_VIEW_KEYS:
            if f'{v}_{key}' not in config:
                errors.append(f"缺少视图键: {v}_{key}")
    return errors
```

## 13. 已知限制与注意事项

### 13.1 已知限制

1. **PnL 孤岛**: 4 个视图的 PnL 完全独立计算，无跨周期信号确认（跨周期 PnL 子图仅是可视化参考）
2. **Dual filter 名实不副**: PnL 仅基于主滤波器，第二滤波器仅做视觉对比
3. **参数统一性**: ke/sm/ew 在所有周期统一，但不同周期波动率量级不同
4. **时间对齐**: 跨周期 PnL 参考使用前向填充，高周期未收盘 bar 的 PnL 尚未反映（显示上一已完成 bar 的值）
5. **无交易日历**: 依赖 Yahoo Finance 数据可用性，无港股特有节假日处理
6. **前视偏差 (Look-Ahead Bias)**: 回测中使用完整信号窗口的抛物线拟合，存在 look-ahead。具体而言：抛物线拟合使用整个 pair 区间的价格数据（含未来点）来确定曲率参数 $a$，拟合时已"看到"了 pair 尾部之后的价格信息。该偏差的量级约为 2-8 个 bar（取决于 `n_extend` 配置），在分钟级数据上影响较小（~数分钟），但在周线/月线上影响可达数周至数月。实盘部署时需将拟合限制在仅使用 pair_end 及之前的数据点

### 13.2 已知 Bug

| # | Bug | 严重性 | 复现条件 | 影响 |
|---|-----|--------|---------|------|
| 1 | **切换标的未重置 offset** | 中 | 在 Sidebar 切换 ticker 后，`day_offset` 保持原值 | 新标的数据窗口错位，可能显示空白图表或错误时间区间 |
| 2 | **同周期 PnL 覆盖** | 中 | 两个视图配置了相同时间框架（如 v0=15分钟, v2=15分钟） | 后渲染视图的 PnL 覆盖先渲染视图的 `session_state` 缓存键，导致跨周期子图引用错误的 PnL 数据 |
| 3 | **无跨周期跳级查找** | 中 | 如 15 分钟视图想查看日线 PnL，但 60 分钟视图未开启策略 | 跨周期子图为空，即使日线 PnL 数据存在 |
| 4 | **compact 参数硬编码** | 低 | `compact=True` 硬编码于 `_render_chart()` | 约 340px 垂直空间未使用（`compact=False` 分支为死代码），图表在分辨率 ≤1080px 时可能溢出 |
| 5 | **配置导入无校验** | 高 | 导入缺失必需键的 JSON 文件 | KeyError 崩溃，而非友好错误提示（已在 §12.5.4 详述） |

### 13.3 分阶段上线路径

从研究平台到实盘交易的推荐上线路径：

```
阶段 A: 离线验证（当前阶段）
  ├─ 历史回测（已实现）
  ├─ Walk-Forward 分析（需实现）
  ├─ 蒙特卡洛模拟（可选）
  └─ 输出: 经时间鲁棒性验证的参数集

阶段 B: 纸交易 (Paper Trading)
  ├─ 实时数据接入（已实现: yfinance → SQLite）
  ├─ 信号生成但不下单（已实现: Schmitt + predict）
  ├─ 模拟成交（需实现: 按 bar 收盘价模拟）
  ├─ 运行周期: ≥ 2 周（覆盖至少一个完整市场周期）
  └─ 输出: 实盘环境下的信号质量报告

阶段 C: 小资金实盘
  ├─ 券商 API 对接（富途 OpenAPI / IBKR）
  ├─ 仓位: 正常仓位的 10-20%
  ├─ 风控: 四层级风控矩阵全量启用（§16）
  ├─ 监控: 实时 PnL + 信号日志（Streamlit 仪表盘）
  ├─ 运行周期: ≥ 1 个月
  └─ 输出: 实盘 PnL vs 回测 PnL 偏差分析

阶段 D: 全量部署
  ├─ 仓位逐步放大（每 2 周 +20%，直到满仓）
  ├─ 风控阈值动态调整（基于实盘数据校准）
  ├─ 自动化运维（定时重启、日志轮转、告警推送）
  └─ 持续监控与月度复盘
```

**各阶段门禁条件**：

| 阶段转换 | 门禁条件 |
|---------|---------|
| A → B | Walk-Forward PnL 三期均为正；最大回撤 < 25% |
| B → C | 纸交易 ≥ 2 周；信号生成无中断；模拟成交率 > 95% |
| C → D | 实盘 ≥ 1 个月；实盘 PnL 与回测偏差 < 50%；无风控违规事件 |

## 16. 四层级风控矩阵

以下风控框架源自 `docs/execution/` 实盘方案，按风险等级分层设计。

### 16.1 风控层级总览

```
Layer 1: 仓位风控 ─── 单笔最大仓位 + 总敞口上限
    │
Layer 2: 止损风控 ─── 硬止损 + ATR 动态止损 + 时间止损
    │
Layer 3: 信号风控 ─── 信号确认 + 周期共振 + 趋势过滤
    │
Layer 4: 系统风控 ─── API 限流 + 连接断开 + 数据延迟
```

### 16.2 Layer 1: 仓位风控

| 规则 | 参数 | 说明 |
|------|------|------|
| 单笔最大仓位 | ≤ 总资金的 5% | 单一交易的风险敞口上限 |
| 总敞口上限 | ≤ 总资金的 30% | 所有持仓合计的最大风险敞口 |
| 同向叠加限制 | ≤ 2 笔 | 同一方向的并发持仓数上限 |
| 跨周期敞口 | 高周期优先 | 高周期信号与低周期信号同向时，低周期减半仓位 |

### 16.3 Layer 2: 止损风控

| 止损类型 | 触发条件 | 优先级 |
|---------|---------|--------|
| **硬止损** | 亏损达入场价的 `stop_loss_pct%`（配置键 `sl`） | 最高 |
| **ATR 动态止损** | 价格跌破 `entry - N × ATR(14)`，其中 N 建议 1.5-2.5 | 高 |
| **时间止损** | 持仓超过 `N` 个 bar 仍未盈利（N 取决于时间框架） | 中 |
| **波动率止损** | 持仓期间波动率突增 3× 入场时波动率 | 低 |

**ATR 动态止损实现建议**：

```python
def atr_stop(entry_price, direction, atr_series, multiplier=2.0):
    """方向感知的 ATR 跟踪止损"""
    if direction == 'long':
        return entry_price - multiplier * atr_series
    else:  # short
        return entry_price + multiplier * atr_series
```

### 16.4 Layer 3: 信号风控

| 规则 | 说明 |
|------|------|
| **双周期确认** | 高周期 Schmitt 信号方向须与本周期一致才入场 |
| **趋势过滤** | EMA(20) > EMA(50) 只做多；EMA(20) < EMA(50) 只做空 |
| **波动率门控** | σ(v) > σ_min 时才允许入场（低波动不入场） |

### 16.5 Layer 4: 系统风控

| 风险 | 应对 |
|------|------|
| API 限流 | yfinance 请求间隔 ≥ 2s，失败重试 3 次（指数退避） |
| 连接断开 | 持仓数据本地缓存，恢复连接后同步 |
| 数据延迟 | 分钟级数据陈旧超过 5 分钟 → 暂停新开仓 |
| 异常报价 | 最新价偏离上一 bar 收盘价 > 10% → 标记为异常，人工审核 |

### 16.6 风控实施状态

| 层级 | 当前状态 | 需开发 |
|------|---------|--------|
| Layer 1 仓位风控 | 未实现 | 需新建仓位管理模块 |
| Layer 2 止损风控 | 部分实现（仅硬止损） | 需添加 ATR/时间/波动率止损 |
| Layer 3 信号风控 | 部分实现（Schmitt 自带滞回过滤） | 需添加双周期确认 + 趋势过滤 |
| Layer 4 系统风控 | 部分实现（API 容错已在 §12.4） | 需添加数据延迟检测 + 异常报价过滤 |
