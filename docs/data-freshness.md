# 数据新鲜度说明

## 问题：日线最后一条 bar 的收盘价偶尔为 nan

### 现象

在某些交易日（尤其是最近 1-2 个交易日），yfinance 获取的日线数据中，最后一条 bar 的 Open/High/Low/Volume 均有值，但 Close 为 nan。导致该 bar 被丢弃，图表显示的数据截止日期比实际滞后 1-2 天。

### 根因

Yahoo Finance 的日线 API 和周线 API 是**两个独立的数据端点，结算时间不同**：

| 端点 | 数据特点 | Close 来源 |
|------|---------|-----------|
| 日线 API (`interval=1d`) | 原始日线，每条独立 | 等待交易所发布官方收盘价。非美股常有 1-2 天结算延迟 |
| 周线 API (`interval=1wk`) | 服务端实时聚合 | 使用实时行情/最新可用价，无延迟 |

这导致一个看似矛盾的现象：同一天的数据，**周线有 Close 而日线没有**。

### 解决方案

在 `_fetch_stock` 中增加了**日线 Close 周线回退机制**（`streamlit_app.py` L514-530）：

```
获取日线数据
  └── 最后 bar Close == nan?
      ├── 否 → 正常处理
      └── 是 → 获取周线数据 → 用周线 Close 回填日线 Close
```

**为什么这个逻辑正确：**

周线 Close = 本周最后一个有效交易日的收盘价 = 日线最后 bar 的实际收盘价。两者逻辑上是同一个值，只是 Yahoo 日线 API 的结算延迟导致日线暂时为 nan。

用周线值回填**不是数据造假**，而是从 Yahoo 自己的周线 API 取回日线 API 尚未结算的值。

### 影响范围

- **适用周期**：仅日线（`interval="1d"`）。周线/月线/季线不受影响。
- **适用市场**：全部市场（A 股、港股、美股）。回退逻辑是市场无关的。
- **回退失败处理**：若周线也无法获取 Close（极罕见），该 bar 仍会被丢弃，行为与修复前一致。

### 其他数据延迟

**DB 缓存过期**（非本修复范围）：

Streamlit App 采用被动刷新架构。数据仅在用户手动操作或自动刷新开启时更新。App 关闭后数据冻结。

- 当前自动刷新默认**关闭**
- 可在侧边栏"数据管理"中手动点击"刷新数据"
- DB 中日线过期 >7 天会在健康检查中告警
