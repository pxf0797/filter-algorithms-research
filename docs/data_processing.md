# 数据处理说明

## 概述

本系统使用 yfinance 作为数据源拉取美股、A股（沪深）、港股的多周期 OHLCV K线数据，以 SQLite 作为本地缓存层，并通过校验、健康检查、快照恢复机制保障数据可靠性。

---

## 1. 数据架构

```
yfinance API
    │
    ▼
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│  _fetch_stock │ ──▶ │  upsert_kline │ ──▶ │  market.db  │
│  (拉取+清洗)  │     │  (写入策略)    │     │  (SQLite)   │
└─────────────┘     └──────────────┘     └──────┬──────┘
                                                │
                                         query_kline()
                                                │
                                                ▼
                                        ┌──────────────┐
                                        │ display/*.parquet │
                                        │ (渲染缓存)       │
                                        └──────────────┘
```

### 1.1 数据存储

- **位置：** `data/market.db`
- **引擎：** SQLite (WAL 模式, `synchronous=NORMAL`, `busy_timeout=5000ms`)
- **表结构：**

```sql
CREATE TABLE kline (
    ticker    TEXT NOT NULL,     -- 股票代码: "AAPL", "3690", "600115"
    timeframe TEXT NOT NULL,     -- 周期: "1分钟", "60分钟", "日线" 等
    ts        TEXT NOT NULL,     -- ISO-8601 时间戳: "2026-06-18T15:30:00+08:00"
    open      REAL,
    high      REAL,
    low       REAL,
    close     REAL,
    volume    REAL,
    PRIMARY KEY (ticker, timeframe, ts)
);
```

### 1.2 写入策略（只追加，不删除历史）

`upsert_kline()` 使用分段策略：

| 数据分类 | 条件 | SQL 操作 | 说明 |
|----------|------|----------|------|
| 历史 K 线 | `ts < DB中MAX(ts)` | `INSERT OR IGNORE` | 已完成的 K 线永不覆盖 |
| 最新 K 线 | `ts >= DB中MAX(ts)` | `INSERT OR REPLACE` | 当前未完成 K 线可更新 |

**关键约束：历史数据不会被自动删除或修改。** 若 yfinance 修正了历史数据（如除权调整），DB 不会自动更新——需通过「数据校验」功能手动触发。

## 2. 数据拉取

### 2.1 周期与参数

| 周期 | yfinance interval | 拉取范围 (period) | 说明 |
|------|------------------|-------------------|------|
| 1分钟 | `1m` | `7d` | 仅最近 7 天 |
| 5分钟 | `5m` | `60d` | 最近 60 天 |
| 15分钟 | `15m` | `60d` | 最近 60 天 |
| 60分钟 | `1h` | `730d` | 约 2 年 |
| 日线 | `1d` | `max` | 全部历史 |
| 周线 | `1wk` | `max` | 全部历史 |
| 月线 | `1mo` | `max` | 全部历史 |
| 季线 | `3mo` | `max` | 全部历史 |

### 2.2 股票代码格式

| 市场 | 示例输入 | yfinance 符号 |
|------|----------|---------------|
| 美股 US | `AAPL` | `AAPL` |
| A股(沪深) | `600115` | `600115.SS`（6开头）/ `000001.SZ`（0/3开头） |
| 港股 HK | `3690` | `3690.HK`（自动补零到4位） |

### 2.3 触发方式

| 触发方式 | 说明 |
|----------|------|
| 首次加载 | 切换 ticker 且 DB 无数据时自动拉取全部 8 周期 |
| 手动刷新 | 侧边栏「刷新数据」按钮，清除缓存后重新拉取全部周期 |
| 自动刷新 | 侧边栏「自动刷新」复选框，按设定间隔（10-600秒）定时拉取 |
| 图表按需 | `_render_chart` 中若 parquet 缓存不可用，单周期拉取（仅 60 天） |

### 2.4 缓存层级

| 层级 | 位置 | TTL | 说明 |
|------|------|-----|------|
| yfinance HTTP | `@st.cache_data` | 300s | Streamlit 会话级缓存，避免重复请求 |
| SQLite | `data/market.db` | 永久 | 跨会话持久化，只追加 |
| Parquet | `data/display/*.parquet` | 永久 | 渲染优化，从 DB 生成 |

## 3. 数据可靠性

### 3.1 功能总览

| 功能 | 位置 | 用途 |
|------|------|------|
| 🩺 数据健康检查 | 侧边栏 expander | 检查 DB 内数据完整性（行数/日期/空值/缺口/过期） |
| 📋 数据校验 | 侧边栏 expander | 对比 DB vs yfinance，发现历史数据修正 |
| 💾 备份与恢复 | 侧边栏 expander | 手动快照、恢复、删除，最多保留 5 个 |
| 📦 导入/导出 | 侧边栏 expander | 跨设备迁移 market.db |

### 3.2 数据健康检查

检查 DB 自身数据的完整性，不涉及外部数据源：

| 检查项 | 方法 | 阈值 |
|--------|------|------|
| 行数 | `SELECT COUNT(*) GROUP BY timeframe` | 0 = 无数据 |
| 日期范围 | `SELECT MIN(date(ts)), MAX(date(ts))` | — |
| 空值 | `SUM(CASE WHEN close IS NULL THEN 1 ELSE 0 END)` | >0 = 异常 |
| 缺口检测 | SQL `LAG()` 窗口函数 | 间隔 > 3倍预期 = 标记 |
| 过期检测 | `julianday('now') - julianday(ts)` | 日线 > 7天 |

**缺口检测详情：**

| 周期 | 预期间隔 | 标记阈值 | 跳过 |
|------|----------|----------|------|
| 1分钟 | 1分钟 | >3分钟 | — |
| 5分钟 | 5分钟 | >15分钟 | — |
| 15分钟 | 15分钟 | >45分钟 | — |
| 60分钟 | 60分钟 | >3小时 | — |
| 日线 | 1天 | >3天 | — |
| 周线/月线/季线 | — | — | ✅ 跳过（间隔不规则） |

> **注意：** 分钟级周期的缺口检测会将夜间/周末休市标记为「缺口」，这是**预期行为**。用户需自行判断哪些是正常的交易时段间隔。

### 3.3 数据校验（DB vs yfinance）

对比 DB 数据与直接从 yfinance 拉取的原始数据，发现两类问题：

1. **新增数据：** yf 有但 DB 没有的 K 线（正常，yfinance 数据更新）
2. **历史修正：** 重叠区间内的 `close` 价格不一致（yfinance 修正了历史数据）

**指纹算法：**

```
重叠区间 = DB.ts ∩ yf.ts
指纹 = MD5(拼接所有重叠 ts 的 "ts:close" 字符串，精确到 4 位小数)
```

| 状态 | 指纹 | 行数 | 含义 |
|------|:--:|------|------|
| `ok` | ✅ | 相同 | 数据一致，无需操作 |
| `update_available` | ✅ | yf 更多 | 仅新增数据，可追加 |
| `conflict` | ❌ | — | 历史数据被修正，需更新 |

**纠错操作：**

- 「更新此周期」按钮 → `force_update_kline()` → DELETE 重叠区间 + INSERT yf 数据 → 清除缓存 → 页面刷新
- `force_update_kline` 保留仅 DB 中存在的非重叠历史 K 线（yf 不再返回但 DB 中有）

### 3.4 备份与恢复

| 操作 | 说明 |
|------|------|
| 创建备份 | 手动触发 → WAL checkpoint → 复制 `market.db` 到 `data/snapshots/market_YYYYMMDD_HHMMSS.db` |
| 恢复 | 下拉选快照 → 覆盖 `market.db` → 清除 WAL/SHM → 清除缓存 → 刷新页面 |
| 删除 | 删除指定快照文件 |
| 自动清理 | 每次创建备份后，保留最新 5 个，删除其余 |

**备份位置：** `data/snapshots/`（已加入 `.gitignore`）

### 3.5 数据库导入/导出

| 操作 | 说明 |
|------|------|
| 导出 | WAL checkpoint → 读取 `market.db` → 浏览器下载（~15MB） |
| 导入 | 上传 `.db` 文件 → schema 校验（确认 kline 表存在）→ MD5 去重 → 覆盖 DB → 清除缓存 → 刷新 |

> **注意：** 导入会直接替换整个数据库文件。如需保留旧数据，导入前先「创建备份」。

## 4. 数据纠错流程

```
怀疑数据有问题
        │
        ▼
  🩺 健康检查
  （检查 DB 自身完整性）
        │
   ┌────┴────┐
   │ 有缺口/空值 │         │ 正常 │
   └────┬────┘         └──┬──┘
        ▼                 ▼
  📋 数据校验          数据可能 OK
  （对比 DB vs yf）
        │
   ┌────┴────┐
   │ 指纹不同 │         │ 指纹一致 │
   │ (conflict)│         │ (ok/update_available) │
   └────┬────┘         └──┬──┘
        ▼                 ▼
  💾 先创建备份         仅新增数据 → 可选更新
        │
        ▼
  「更新此周期」
  force_update_kline()
        │
        ▼
  页面自动刷新 → 数据已修正
```

## 5. 已知问题与注意事项

### 5.1 yfinance 数据不可重复性

yfinance 在不同时间拉取同一 ticker+周期 的历史数据，可能返回**略微不同**的结果：
- CDN 节点差异
- 数据修正/复权调整
- 日内数据的时间戳可能略有偏移

**缓解方案：** 使用「📋 数据校验」定期检查，发现差异后通过 `force_update_kline` 更新。

### 5.2 缺口检测误报

夜间休市、周末、节假日都会在分钟级周期产生 >3倍预期间隔的缺口。这些是**正常现象**，不代表数据有问题。健康检查标记缺口是为了**让人看到全貌**，而非报警。

### 5.3 数据追加假设

`upsert_kline` 的 `INSERT OR IGNORE` 策略假设：**一旦 K 线生成，历史数据不会再变化**。这个假设在 yfinance 场景下不完全成立（除权调整可能导致历史数据变化）。如果发现数据修正，需通过「数据校验」手动更新。

### 5.4 60分钟数据范围

批量刷新使用 `period="730d"`（约 2 年），但图表按需拉取使用 `period="60d"`（仅 60 天）。如需更长历史，使用「刷新数据」按钮。
